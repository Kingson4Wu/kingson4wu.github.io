使用Java调用Rust构建的动态库

2025年3月23日
  • Java
  • Rust
  • Programming Languages

近期帮朋友做一个需求，需要将某个 Rust 项目的一部分功能移植到 Java 中，使 Java 应用能使用

这次需求本质上是：已有 Rust 项目里有一部分能力，Java 应用也想复用。

一开始的直觉是把 Rust 代码翻译成 Java。但实际试了一下之后，发现这条路不划算，最后改成把 Rust 功能编译成动态库，再由 Java 调用。

方案一：把 Rust 翻译成 Java

理论上这是最干净的方案。Java 项目不需要 native 依赖，代码也更容易统一维护。

前置工作是先把 Rust 项目跑通，最好能跑通相关测试用例，再通过 debug 对照执行细节，一点点翻译成 Java。

我尝试借助 AI 做这件事：

  1. 先跑通需要抽取功能的测试用例。
  2. 让 AI 帮忙把相关代码抽成一个单独 Rust 项目。
  3. 再让 AI 按模块把 Rust 代码翻译成 Java。

实际效果一般。

主要问题是上下文太多后，AI 很容易抽错边界，生成出来的代码也比较乱。后来我改成自己拆分代码块，让 AI 逐段转换，效果好一些，但效率又太低，而且仍然需要人工慢慢修。

这次的结论是：AI 对小范围代码迁移有帮助，但一次性迁移大量、带复杂上下文的代码，还不能完全托管给它。

所以我放弃了完整翻译。

方案二：Rust 编译动态库，Java 直接调用

更现实的方案是保留 Rust 实现，把相关功能编译成动态库，由 Java 通过 JNI 调用。

这样做的好处很直接：

  1. 不需要重写核心逻辑。
  2. Rust 原有测试和实现可以继续复用。
  3. 避免翻译代码带来的隐性 bug。
  4. Java 侧只需要处理加载、调用和平台兼容。

代价也很明确：

  1. 需要为不同平台构建动态库，比如 Windows 的 .dll、Linux 的 .so、macOS 的 .dylib。
  2. Java 加载前要判断当前系统和架构，选择对应库文件。
  3. 需要测试 Java 8、Java 9 以上模块化系统，以及不同操作系统的加载行为。
  4. 构建发布流程会比纯 Java 项目复杂。

这类复杂度可以用 GitHub Actions 解决一部分。不同平台自动构建动态库，再随 Java 包一起发布。

JNI 调用的性能判断

Java 通过 JNI 调用 Rust 动态库，发生在同一操作系统进程、同一虚拟地址空间内。动态库加载后被映射到进程地址空间，后续调用不是重新启动进程，也不是额外发起网络请求。

普通 Java 方法在热点场景下会被 JIT 编译成本地机器码，JNI 调用进入 Rust 后执行的也是 Rust 编译器生成的机器码。从“是否执行机器码”这个角度看，两者没有本质差异。

真正的差异在边界。

Java 方法可以被 JVM 内联和优化，很多调用开销可以被消掉；JNI 调用对 JVM 来说是一条穿不过去的边界，不能内联，每次调用都要付出固定的边界维护成本。

这个成本通常不是系统调用级别的大开销，但也不是零。是否划算，取决于每次 JNI 调用里 Rust 做的工作是否足够多。

如果只是调用一个极小函数，频繁跨 JNI 边界，可能不划算；如果一次调用里完成较重的计算或完整业务逻辑，JNI 的固定成本就可以被摊平。

所以可以简单理解为：

JNI 的边界成本固定且不可消除，但 Rust 代码本身不会因为被 Java 调用就变慢。

小结

这次我最后选择动态库方案，不是因为它绝对优雅，而是因为它在当前场景下更省事、更稳。

如果 Rust 逻辑比较小、业务长期要在 Java 中维护，翻译成 Java 可能更合适；如果 Rust 逻辑已经成熟、迁移成本高、性能或正确性比较重要，那保留 Rust，通过 JNI 调用动态库是一个更现实的方案。
